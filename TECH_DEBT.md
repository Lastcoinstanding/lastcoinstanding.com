# Tech Debt — Last Coin Standing

A running list of known issues, inconsistencies, and architectural improvements identified during ongoing work but deferred from immediate sessions. Items here are deliberately not blocking to ship — they're tracked so we can prioritize and address them in dedicated cleanup passes rather than losing them in chat history.

**Guiding principle:** consistency-by-default across all pages and all new pages; introduce inconsistencies only when there's a clear rationale. Items below are usually ones that violate this principle, or that capture a strategic decision worth making deliberately rather than drive-by.

## Conventions

- **Status:** `- [ ]` open, `- [x]` closed
- **When closing:** move to "Recently closed" with the commit SHA
- **When adding:** keep entries terse; deep context lives in the relevant commit message (`git log --grep` works well)
- **Bundles:** when one item is a sub-task of another, indent under the parent

- [x] **not-a-bubble.js DM Sans → Inter swap — residual from §5.7 sweep (commit `808e50f`).** Closes the chart-label rendering loose end. 26 hardcoded `'DM Sans', sans-serif` and `'DM Sans', -apple-system, sans-serif` references in Chart.js options and canvas ctx.font declarations were swapped to Inter via mechanical replace. Audit before swapping revealed this was a strict improvement, not a judgment call: DM Sans is NOT loaded on Not-a-Bubble (no @font-face, not in the Google Fonts URL — the §5.7 CSS sweep removed it), so all 26 JS references were already falling back to the second item in each fallback stack — i.e., the browser's system sans-serif default. That's Segoe UI on Windows, -apple-system on Mac, Roboto on Android — cross-platform inconsistency, none matching the page's body font. After the swap, chart labels render in Inter consistently across platforms, matching the page's editorial register per STYLE_GUIDE §1. The decision criteria in the original entry ("do the chart labels visibly degrade if swapped?") was answering the wrong question: the labels weren't rendering in DM Sans before. Process implication: when font-family references appear in JS for canvas/chart rendering, also check whether the named font is actually loaded by the page; absent loading, the JS reference is silently using fallback, which is itself a cross-platform inconsistency worth fixing. - [x] **Disciplined Rebalancing channel-viz tooltip + scale-cache bug — multi-commit chain ending `535626a`.** Long debugging arc on the channel viz; final state works correctly across desktop/mobile, hover at any cursor x shows the right tooltip values with active-point indicators tracking the cursor, and the historical price line ends right next to the Today line at any horizon-slider value. Worth capturing at length because the diagnosis took several wrong turns and the underlying lessons are subtle enough to bite again on any future chart-in-tab page.

  **The visible bugs** (in order of how they surfaced): (1) tooltip body blank on hover, only active-point circles visible; (2) tooltip body correct but circles drawn far from the cursor; (3) historical price line ending visually at year ~2021 even though its last data point is Apr 2026 (the line *should* end right next to the Today line at year 2026.4). All three turned out to be different symptoms of two intertwined root causes.

  **Root cause 1 — Chart.js v4's `mode: 'index'` index-mismatch.** The original tooltip mode was `interaction: { mode: 'index', axis: 'x' }`. With `axis: 'x'`, Chart.js finds the dataset whose nearest-in-x point is closest to the cursor, then applies that point's *index* to all datasets. Bands sample every 30 days from minD to maxD; historical price uses PL_DATA's ~12-day cadence, which is locally denser in the historical range. So when hovering in the historical range, Chart.js picks historical-price's index, then asks "what's at index N?" of every dataset. Bands have 1300+ points spanning 2010-2046; bands[N] (where N comes from a 477-point dataset) lands at a year wildly different from the cursor. Result: active-point circles render at the wrong x; tooltip rows show data from inconsistent positions. Fixed by registering a custom interaction mode `xPerDataset` (`6932f08`) that iterates each dataset independently, finds the closest-x element per dataset by *pixel proximity* (not data-space distance — the latter looked right but produced misaligned circles when element pixel positions were stale), and returns one item per dataset within a 15-pixel tolerance.

  **Root cause 2 — Chart.js v4's layout cache survives `chart.update('none')`.** The horizon slider mutates band data (different array refs, different lengths) and triggers a scale-max recompute. Default expectation: every controller's `updateElements` runs against the new scale, every element's pixel position recomputes. Empirical reality: the layout cache from the chart's *first* render survives, so historical-price elements (whose data array reference didn't change across the horizon-slider event) keep pixel positions baked against the original scale. Last historical-price point at day 6304 stays drawn at the pixel for day 6304 *under the original scale*, not the new one. The visual symptom is the historical line ending hundreds of pixels from where it should. Fixed by calling `chart.update('resize')` after the data mutation (`a3acd70`) — the only update mode that invalidates the layout cache without causing a flicker.

  **Root cause 3 (compounding) — sticky values fires the horizon-input event during init while the tab is hidden.** Sticky values restoration runs at the end of the page-init IIFE chain. If the user has a stored `dr:horizon` value and arrives at the page without `#calculator` in the URL, the calculator tab is `display: none` at that moment — `updateXDomain` runs against a 0×0 canvas, and `chart.update('resize')` reads the zero dimensions and corrupts every element to pixel 0. Worse than the no-fix state. Final fix: `pendingLayoutFix` flag + `ResizeObserver` on the canvas, so the layout-cache invalidation defers until the canvas has real dimensions (i.e., until the user clicks into the tab). Pattern documented as STYLE_GUIDE §6.14 for adoption on any future chart-in-tab pages.

  **Tooling that unstuck the diagnosis.** Several intermediate fixes shipped that worked in puppeteer but failed for the user, because the puppeteer tests didn't replicate the exact init order (sticky-values seeding `dr:horizon` *before* page load + chart inside hidden `.tab-content`). The breakthrough was building a puppeteer harness with `evaluateOnNewDocument` to seed `localStorage` *before* navigation, then comparing element pixel positions to `xs.getPixelForValue(rawData[i].x)` for in-sync vs stale. That harness reproduced the user's exact state in <5 seconds and made it possible to A/B different update modes (`'none'` / default / `'reset'` / `chart.resize()` / `chart.update('resize')`) against the broken state in a single page-evaluate. Lesson: when a remote-debugging fix appears to work in dev but not in production, the dev test is missing init-order details — building a harness that seeds the user's *exact* initial state (including pre-navigation localStorage) is faster than re-deploying iterative guesses.

  **Surface-area scan after fix.** Audited the rest of the site for charts inside `display: none` ancestors at script-execution time. Disciplined Rebalancing is currently the only page where this combination happens (chart inside `.tab-content` + sticky values that mutates chart-affecting state at init). Other tab-bearing pages (Half-Life, Power Law, MIC, Fixed Pie) either have their charts in always-visible tabs or don't have init-time data mutation. New chart-bearing pages adopting the `.tab-content` system should consult §6.14.

  **Process improvement banked.** When a chart bug presents as "data is right, rendering is wrong," the next diagnostic move is: read `meta.data[i].x` (element pixel) and compare to `chart.scales.x.getPixelForValue(rawData[i].x)` (what the current scale says it should be). If they disagree, the layout cache is stale; the fix is `chart.update('resize')` on the path that mutated the data. If they agree but the visual is still wrong, the bug is in the data or the interaction mode, not the cache. This single check would have collapsed the multi-commit diagnostic chain to one or two iterations.
